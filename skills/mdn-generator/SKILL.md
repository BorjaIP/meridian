---
name: mdn-generator
model: medium
description: >
  Meridian Agent Loop task generator. Reads the current backlog, recently
  completed tasks, `REVIEW.md` feedback, and either `GOAL.md` (goal
  mode) or the remaining iteration count (iterations mode), and appends new
  `owner::agent status::backlog` tasks to the project note so the loop always
  has something to do. Use when the user says "generate new tasks", "fill the
  backlog", "run the generator", or when invoked by the Agent Loop orchestrator
  via `/mdn-generator`.
---

## Role

This skill is one of the three autonomous roles of **Agent Loop Mode** (see
`docs/agent-loop.md`). Its only job is to keep the backlog populated with
high-quality, goal-aligned tasks so the Reviewer/Runner pair never starves.

## Invocation

```
/mdn-generator project:<slug> mode:<iterations|goal> [target:<n>] [loop-id:<id>]
```

| Argument | Required | Default | Description |
|---|---|---|---|
| `project` | yes | — | Meridian project slug. |
| `mode` | yes | — | `iterations` or `goal`. Determines the input signal. |
| `target` | no | `3` | Desired number of new tasks to emit in this invocation. |
| `loop-id` | no | — | Loop identifier from `mdn-loop`. When present, stamp each generated task with `loop-id::<id>` on its checkbox line. |

## Execution steps

### Step 0 — Resolve config
Read `${XDG_CONFIG_HOME:-~/.config}/meridian/config.md`. Extract `vault`. Run the
standard date generation snippet → `<NOW>`. If missing: tell user to run
`/mdn-init`.

### Step 1 — Find project note
Read `<vault>/meridian/<slug>/PROJECT.md`. Verify `project: <slug>` in
frontmatter. If missing: report "Project `<slug>` not found" and stop.

### Step 2 — Read all signals
Collect, in order:

1. **Backlog** — every `owner::agent status::backlog` task block.
2. **Recently done** — every `owner::agent status::done` task block (full list;
   the most recent ones are usually the relevant signal).
3. **Blocked** — any `status::blocked` tasks. These are gaps the Generator must
   not reopen.
4. **REVIEW.md** — read `<vault>/meridian/<slug>/REVIEW.md` if it exists.
   Parse the most recent Reviewer blocks. Pay special attention to:
   - `Next action: create correction task: ...` entries (already-open work).
   - The **last `## GOAL CHECK` block's `Next focus:` line** (goal mode only).
   - `needs-correction` verdicts — these reveal recurring weak spots.
5. **Objective signal:**
   - `mode:goal` → read `<vault>/meridian/<slug>/GOAL.md` and extract the
     active goal (status: active). If none: print "No active goal in GOAL.md."
     and stop.
   - `mode:iterations` → read `## Iterations` in the project note and extract
     `Remaining`. If missing: print "No `## Iterations` section in PROJECT.md."
     and stop.
6. **Project note body** — the Summary / Scope / Notes sections at the top of
   the project note, so tasks stay on-theme.

### Step 3 — Decide what's missing
Think like a lead engineer planning the next mini-sprint. Do not pad. Do not
duplicate open work. Aim for tasks that:

- Move **concretely** toward the active goal (goal mode) or fill natural gaps in
  the project (iterations mode).
- Are **independent** of each other where possible — the orchestrator may run
  them in any order.
- Are **self-contained** — each task has enough context in its Description and
  Acceptance that `mdn-run` in simple flow (Mode B) can execute it without a
  plan. For genuinely large work, mark `type::feature priority::high` so the
  orchestrator knows to spawn `mdn-plan` first.
- Respect the **Next focus** hint from the last GOAL CHECK block verbatim if one
  exists.

Rules:
- **Never re-emit** a task whose title is within 90% similarity of an existing
  backlog or blocked title.
- **Cap at `target`** new tasks per invocation. Fewer is fine — it is better to
  emit one excellent task than three mediocre ones.
- In iterations mode, if `Remaining <= 1`, emit at most one task (a single
  wrap-up / polish task).
- If nothing meaningful is missing, print "Backlog is sufficient — no tasks
  generated." and stop. This is a valid outcome.

### Step 4 — Write the tasks
Append each new task to the `## Tasks` section of `PROJECT.md` using the exact
Meridian task format (matches `templates/Task.md` and the shape emitted by
`/mdn-add`):

```markdown
- [ ] #task owner::agent status::backlog type::<feature|fix|research|chore> priority::<high|medium|low> [loop-id::<id>]
  **Title:** <short imperative title>
  **Description:** <1–4 lines: what and why, grounded in the signals read in Step 2>
  **Acceptance:** <what "done" looks like, specific enough to be verifiable by the Reviewer>
  **Generated by:** mdn-generator on <NOW> (<mode> mode)
```

Include `loop-id::<id>` on the checkbox line only when the `loop-id:` argument was
supplied. Omit the field entirely when running standalone.

Choose `type` and `priority` based on the task's nature:
- `feature` — new capability; default `priority::medium`.
- `fix` — repairs a `needs-correction` finding; default `priority::high`.
- `research` — exploration with unclear outcome; default `priority::medium`.
- `chore` — maintenance, docs, tidy-up; default `priority::low`.

Upsert each new task into the `## Tasks` summary table at the top of the section
if that table exists, with status `backlog`.

### Step 5 — Print a summary
```
✓  Generated <n> task(s) for `<slug>` (<mode> mode).
   Signals used: backlog=<b> done=<d> review-blocks=<r> goal=<"..."|remaining=<k>>
   New tasks:
     1. <title>  [type::...  priority::...]
     2. ...
```

### Step 6 — Output contract
Always end with a single line of machine-readable output for the orchestrator:

```
generator-result: generated=<n> skipped_reason=<none|sufficient|no_goal|no_iterations>
```

## Meridian protocol reference

- Tasks emitted by this skill are indistinguishable from tasks written by
  `/mdn-add`. Every other skill (`mdn-run`, `mdn-plan`, `mdn-reviewer`,
  `mdn-status`, `mdn-daily`) works on them unchanged.
- The Generator is **read-only** outside `PROJECT.md`. It never writes to
  `REVIEW.md` or `GOAL.md` — only the Reviewer does.
